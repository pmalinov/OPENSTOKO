## 1. Какво е OPENSTOKO
OPENSTOKO е система за склад:
1. Въвеждаш продукти.
2. Следиш наличности.
3. Правиш вход и изход на стока.
4. Виждаш алтернативни продукти (заместители).
5. Виждаш история кой какво е направил.

## 2. Вход в системата
1. Отвори браузър.
2. Въведи адрес: `http://localhost:3000`
3. По подразбиране езикът е български. Можеш да го смениш горе вдясно.
4. Вход с готови потребители:
5. При чисто нова инсталация винаги има:
6. `admin` / `admin123`
7. `operator1` се създава от админ панела при нужда.

## 3. Разлика между ролите
1. `Админ`:
2. Може да създава категории.
3. Може да редактира цени и минимални прагове.
4. Има достъп до отчети, анализи, история, админ панел.
5. `Оператор`:
6. Работи бързо по приемане/изписване.
7. Вижда праг, но не може да го променя.
8. Може да праща сигнал към админ.

## 4. Меню и основни екрани
### 4.1 Екран „Оператор“
1. Режим `Приемане` (вход на стока).
2. Режим `Изписване` (продажба/изход).
3. Начин на търсене при изписване:
4. По баркод (по подразбиране).
5. По вътрешен код.
6. По име.
7. По категория.

### 4.2 Екран „Търсене“
1. Отваряш продуктово досие.
2. Виждаш снимка, наличност, праг, статус, коментари, алтернативи.
3. При админ има и admin действия.

### 4.3 Екран „Добавяне на продукти“ (админ)
1. Създаваш нов продукт.
2. Виждаш таблица с продукти и здраве на наличностите.
3. Можеш да редактираш минимален праг (админ).

### 4.4 Екран „Инвентаризация“
1. Показва `Критично`, `Предупреждение`, `Нормално`.
2. Можеш да кликнеш ред на продукт за детайл.

### 4.5 Екрани „Анализи“, „Отчети“, „История“, „Админ“
1. Анализи: Time Machine, Velocity, ABC, гаранция.
2. Отчети: Excel експорти.
3. История: одит следа кой какво е направил.
4. Админ: предложения за праг, потребители, системни настройки.

### 4.6 Пълна карта на бутоните (какво става при клик)
#### 4.6.1 Операторски екран (бутон по бутон)
1. `Смени режим`: превключва между `Приемане` и `Изписване`.
2. `Търси`: показва резултати според избрания метод (баркод/вътрешен код/име/категория).
3. `Избери` в таблицата: добавя конкретния ред в текущата продажба (cart).
4. `Потвърди вход`: записва вход на стока; при auto режим генерира SN за всяка бройка.
5. `Потвърди продажба`: финализира checkout за всички редове в cart и създава обща референция.
6. `Принт гаранция`: сваля PDF за конкретен SN.
7. `Принт всички`: сваля/пуска печат на гаранционни етикети за всички SN от текущата продажба.
8. `Сигнал към админ`: създава нотификация за админ (без промяна на праг/цени).
9. `Изход от Оператор`: чисти текущата работна сесия на екрана (не logout).
10. Горни бутони:
11. `Помощ`: отваря вградени инструкции.
12. `Обнови`: презарежда данните от backend.
13. `Изход`: logout от системата.

#### 4.6.2 Админски менюта вляво (къде влизаш и какво правиш)
1. `Оператор`: отваря операторския работен поток (приемане/изписване).
2. `Добавяне на продукти`: създаване на нов продукт, импорт от Excel, админ редакция на продукти.
3. `Инвентаризация`: общ преглед на наличности и статуси (`Критично/Предупреждение/Нормално`).
4. `Анализи`: BI екрани (Time Machine, Velocity, ABC, гаранция по SN).
5. `Отчети`: Excel експорт на продукти, движения и одит.
6. `История`: одит логове с филтри по обект, потребител и сериен номер.
7. `Сторно одобрения`: чакащи заявки за сторно с бутони `Одобри` / `Откажи`.
8. `Помощ`: пълно описание на работните потоци.
9. `За OPENSTOKO`: информация за версия, лиценз и контакти.
10. `Админ`: създаване потребители, reset пароли, настройки на сесия и предложения за праг.

#### 4.6.3 Админ екран „Добавяне на продукти“ (под-бутони)
1. `+ Добави нова категория`: включва режим за създаване на нова категория в самата форма.
2. `Създай категория`: записва въведената категория и я избира автоматично.
3. `Нов продукт`: валидира задължителните полета и създава продукт.
4. `Импорт Excel`: качва Excel и създава/обновява продукти според файла.
5. В таблица `Продукти`:
6. `Редакция`: отваря панел за админ редакция на всички продуктови полета (без SN на единични бройки).
7. `Запази` (до `Мин. праг`): записва нов минимален праг за избрания продукт.
8. `Баркод`: сваля PNG баркод.
9. `Етикет PDF`: сваля PDF етикет за печат.

#### 4.6.4 Админ екран „История“
1. Поле `Обект`: филтрира по тип действие (`product`, `category`, `stock_movement`, `auth`, `user`).
2. Поле `Потребител`: филтрира по username.
3. Поле `Сериен номер`: филтрира логовете по SN и помага за гаранционна проследимост.
4. `Зареди история`: презарежда таблицата с текущите филтри.
5. `Провери гаранция`: прави директна гаранционна проверка за въведения SN.
6. `Инфо`: отваря детайлен модал със старите и новите стойности на записа.

## 5. Как да въведеш продукт (стъпка по стъпка)
1. Влез с `admin`.
2. Отиди в `Добавяне на продукти`.
3. Ако категорията липсва:
4. Създай категория от секцията за категории.
5. Попълни:
6. Име (задължително).
7. Категория (задължително).
8. Фабричен баркод EAN/UPC (задължително).
9. Вътрешен баркод (по желание).
10. Доставчик (по желание).
11. Локация (задължително).
12. Покупна цена (задължително).
13. Продажна цена (задължително).
14. Минимална продажна цена (задължително).
15. Минимален праг (задължително).
16. Съвместимост (`compatibility group`) ако има заместители.
17. Запази.
18. Ако полето за вътрешен код е празно, системата автоматично генерира SKU във формат `PREFIX-ГОДИНА-ID`.

Важно:
1. Фабричният баркод (EAN/UPC) е код на модела/продукта.
2. Серийният номер (SN) е уникален номер за конкретна физическа бройка и се създава/ползва при вход и изход на стока.

## 6. Как да направиш приемане (вход) на стока
1. Влез като оператор или админ.
2. Отиди на `Оператор`.
3. Избери режим `Приемане`.
4. Сканирай фабричен/вътрешен баркод или вътрешен код на продукта.
5. Въведи количество.
6. Натисни `Потвърди вход`.
7. Системата създава уникални серийни номера (ако е в auto режим).

## 7. Как да направиш изписване (продажба)
1. Отиди на `Оператор`.
2. Избери режим `Изписване`.
3. Избери метод за търсене:
4. `Търси по баркод`: сканирай код.
5. `Търси по вътрешен код`: търси по вътрешния код на продукта.
6. `Търси по име`: въведи име/марка и натисни `Търси`.
7. `Търси по категория`: избери категория от падащото меню и `Търси`.
8. При списък с резултати кликни на желания продукт.
9. Детайлът се отваря веднага под избрания ред.
10. Въведи количество и клиент.
11. Натисни `Потвърди продажба`.
12. След успешна продажба в горната част се показват продадените серийни номера (SN).
13. Отваря се специален панел `Екран за печат на гаранционни етикети`.
14. Има бутон `Принт всички` и отделен бутон `Принт гаранция` за всеки SN.
15. Натисни бутона и системата сваля PDF етикет за гаранцията (с баркод на SN).

## 7.1 Как да залепиш гаранционен етикет
1. След продажба натисни `Принт гаранция` до желания SN.
2. Отвори сваления PDF (`warranty_<SN>.pdf`).
3. Принтирай етикета и го залепи към гаранционната карта/документ.
4. При рекламация сканирай баркода на SN от етикета.
5. Провери гаранцията в `Анализи -> Гаранция` с този SN.

## 8. Как работи търсенето по категория
1. Показва продукти само от избраната категория.
2. Можеш да кликнеш върху ред за детайл.
3. В детайла виждаш:
4. Наличност.
5. Праг.
6. Статус.
7. Продажна цена.
8. Алтернативи/заместители.

## 9. Статуси на наличността
1. `Критично`: наличност `<=` праг.
2. `Предупреждение`: наличност `<=` праг x 1.2.
3. `Нормално`: наличност над warning зоната.

## 10. Заместители и алтернативи
1. Ако продуктите са в една и съща compatibility група, системата ги показва като алтернативи.
2. Пример: различни марки накладки за една и съща кола.
3. Пример извън авто: RAM `DDR4_3200MHz_16GB` от различни марки.

## 11. Как операторът сигнализира админ
1. В детайла на продукта операторът вижда бутон `Сигнал към админ`.
2. Натисни бутона.
3. Можеш да добавиш кратка бележка.
4. Админ получава нотификация.

## 12. История и проследимост
1. Всяко важно действие се записва в одит следа:
2. Кой.
3. Кога.
4. От кой IP.
5. Какво е променено.
6. Движенията на склада се пазят като ledger (история).

## 13. Важно за сигурността на одита
1. В приложението няма функция за триене на одит записи.
2. На среди с нужните DB права се добавя защита в MySQL: `audit_logs` става immutable (блокирани `UPDATE/DELETE`).
3. Ако средата не позволява trigger-и, остава app-level защита и одит следа.
4. Забележка: ако някой има директен root достъп до MySQL сървъра, това е извън приложението.

## 14. Полезни ежедневни сценарии
### 14.1 Бърз прием
1. Оператор -> Приемане -> скан -> количество -> потвърди.

### 14.2 Бърза продажба по категория
1. Оператор -> Изписване -> Търси по категория -> избери категория -> Търси -> клик на продукт -> количество -> потвърди.

### 14.3 Проверка на критични продукти
1. Инвентаризация -> филтър `Критично`.
2. Клик на реда за продуктово досие.
3. При нужда сигнал към админ.

## 15. Чести проблеми и решения
1. Няма резултати в търсене:
2. Провери дали режимът на търсене е правилен (баркод/код/име/категория).
3. Провери правопис и интервали.
4. Няма последни продажби:
5. Няма направени `output` движения. Направи тестова продажба.
6. Не можеш да редактираш праг:
7. Това е нормално за оператор. Само админ може.

## 16. Адреси и достъп

2. API docs: `http://localhost:8000/docs`
3. phpMyAdmin (ако е стартиран): `http://localhost:8081`

## 17. Ако забравиш парола за admin
1. Отвори `.env`.
2. Задай:
3. `RESET_ADMIN_PASSWORD_ON_START=true`
4. `BOOTSTRAP_ADMIN_PASSWORD=admin123`
5. Рестартирай backend.
6. Влез с `admin/admin123`.
7. Върни `RESET_ADMIN_PASSWORD_ON_START=false`.

## 18. Пълен reset на данните и ново зареждане (само системен администратор)
Това НЕ е ежедневна операция. Ползвай само за тестова среда или чист старт.

### 18.1 Изтриване на всички данни
1. Отвори terminal в папката на проекта.
2. Изпълни:
```bash
docker compose exec backend python -m app.scripts.wipe_all_data
```
3. Това изчиства бизнес данните и пази текущите потребители.
4. Скриптът никога не пипа таблица `users`.
5. `admin` login и паролата на `admin` остават както са.

### 18.2 Зареждане на реалистични тестови данни
1. Изпълни:
```bash
docker compose exec backend python -m app.scripts.seed_simulation
```
2. Скриптът създава:
3. Категории и продукти.
4. Сериен номер за всяка физическа бройка.
5. Вход/изход движения за 1 година назад.
6. Дефекти, инвентарни корекции, сторно заявки.
7. Гаранционни SN за печат и проследяване.

### 18.3 По желание: seed за друг период
```bash
docker compose exec backend python -m app.scripts.seed_simulation --days 540 --seed 20270101
```
