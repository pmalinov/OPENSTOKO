# OPENSTOKO Потребителско Ръководство (Български)



OPENSTOKO е система за склад и продажби с ролеви достъп, изградена върху FastAPI + Next.js + MySQL 8.4.

## Важно уточнение (Правен дисклеймър)
- Проектът е open-source и е предназначен за вътрешен складов контрол и управление на наличности.
- Софтуерът **не е СУПТО**, **не е свързан с НАП** и **не покрива изискванията на Наредба Н-18** за фискализация.
- Не използвайте приложението като единствен фискален/касов софтуер за регулаторни цели без допълнителна сертифицирана интеграция.
- Авторът не носи отговорност за глоби, санкции или финансови щети при неправилна употреба за фискални цели.

## 1. Достъп
- Приложение: `http://localhost:3000`
- API документация: `http://localhost:8000/docs`
- phpMyAdmin (по избор): `http://localhost:8081`

Потребители по подразбиране (чиста инсталация):
- `admin` / `admin123`
- `operator1` се създава от админ панела при нужда

## 2. Роли
- `operator`: ежедневен вход/изход на стока и checkout.
- `admin`: управление на продукти, цени, прагове, потребители, Data Management (импорт/експорт), сторно одобрения.

## 3. Езици
- Английски (`en`)
- Български (`bg`)
- Италиански (`it`)

## 4. Операторски поток
1. Отвори `Оператор`.
2. Избери режим: `Приемане` или `Изписване`.
3. При изписване търсенето е с приоритет:
- фабричен баркод
- вътрешен код (SKU)
- име
- категория
4. Добави артикули в чернова (cart).
5. Провери междинни суми и крайна сума.
6. Потвърди продажбата.
7. Принтирай гаранционни етикети за продадените SN.

## 5. Админ поток
В `Добавяне на продукти` админът редактира продуктовите параметри.
Серийните номера са уникални за физическите бройки и не се редактират от продуктовата форма.

Подробно описание бутон-по-бутон (всички менюта и под-бутони) е в:
- `docs/technical/README_BG_USER_GUIDE.md` -> секция `4.6 Пълна карта на бутоните`.

## 6. Inventory Health
- `Критично`: наличност <= праг
- `Предупреждение`: наличност <= праг * 1.2
- `Нормално`: над warning зоната

## 7. Сторно одобрения
- Оператор подава заявка за сторно.
- Админ одобрява/отказва.
- При одобрение серийният номер се връща в наличност и остава в историята.

## 8. Одит и сигурност
- Важните действия се записват в `audit_logs`.
- Админ редакции пазят стари/нови стойности.
- Няма endpoint за триене на одит.
- При позволени DB права има trigger защита срещу UPDATE/DELETE върху `audit_logs`.
